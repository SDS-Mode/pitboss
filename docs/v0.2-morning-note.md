### Real v0.1 bug caught while building the TUI
While debugging why the TUI showed all tasks as "Pending" for completed runs, I
discovered that **`summary.jsonl` is empty for every run we ever produced**.
The spec's incremental-summary invariant (§5.3) requires each task's record to
fsync to `summary.jsonl` on completion — so a mid-run kill still leaves per-task
data on disk. The v0.1 dispatch runner **never called `store.append_record`**;
it only called `finalize_run` at the end. Net effect: `summary.json` was
written correctly on clean exit, but `summary.jsonl` was always 0 bytes, and a
`kill -9` during a run would have lost all completed-task state. Fixed in
commits `5845f91` (append_record wired into `execute_task`) and `0387276`
(regression test that fails the whole suite if anyone ever drops the call again).

The Mosaic TUI watcher was also updated to read `summary.json` as the primary
source with `summary.jsonl` as the fallback for in-progress runs — otherwise
historical runs (produced before the fix) would render as all-Pending.

### Bonus: `pitboss-tui screenshot`

## Known quirks

- The `duration_ms` in the status bar is the **sum** of per-task durations
  (CPU-time style), not wall-clock. With parallel tasks this overcounts wall-clock
  but is more informative about total compute. Documented inline in the
  `run_stats` function.
- Non-ASCII emoji were avoided in the focus pane markers (`>`, `*`, `<`, `v`, `!`)
  to sidestep any terminal width weirdness. Looks fine on xterm/alacritty/wezterm.
- The top-level README now says `cargo install --path crates/pitboss-tui` works.
  That's true but `--release` is the flavor you want for the TUI.
