# Agent Shire v0.2 — Plan (provisional)

**Authored:** 2026-04-16, overnight work session
**Status:** in-flight; expect `feat/v0.2` branch to grow during this document's lifetime

## Goals

Deliver a usable **Mosaic TUI v0.2-alpha** on top of the shipped v0.1 foundation. The
headless `shire` CLI already produces a well-structured run directory under
`~/.local/share/shire/runs/`. The TUI's first job is to **observe** those runs live —
no launch-from-TUI, no snap-in/out, no SQLite resume. That scope keeps this achievable
in one overnight session; richer interaction lands in v0.2.1+.

## In scope for v0.2-alpha

- A new `mosaic` binary (workspace crate `mosaic-tui`)
- Reads any active or past run dir under `~/.local/share/shire/runs/`
- Tile grid: one tile per task, colored by status
- Real-time state updates via filesystem watch or short-interval poll of `summary.jsonl`
- Focus pane: selected tile's last ~40 lines of `stdout.log`
- Keybindings: `h j k l` / arrows navigate, `L` view full log, `r` refresh, `q` quit,
  `?` help
- Works against runs created by v0.1 shire without any changes to shire

## Deferred (v0.2.1+)

- Snap-in interactive mode (keystroke passthrough to live session)
- Spawning new sessions from within the TUI
- `SqliteStore` backing + `claude --resume` integration
- Cross-restart session attach
- Broadcast mode

## Loose-end fixes bundled in v0.2 (pre-TUI)

Small, mechanical improvements that make the v0.1 core more observable for the TUI:

1. **Wire stderr → `stderr.log` per task.** Currently dropped. Would have exposed the
   `--verbose` silent failure immediately if wired during v0.1.
2. **Explicit `RateLimitEvent` variant in the parser.** Currently falls into `Unknown`.
   Useful for operator visibility.
3. **Smarter `final_message_preview`.** Track the longest-nontrivial assistant text
   rather than the literal last one — avoids "Done." winning the slot.

## Crate layout after v0.2-alpha

```
crates/
├── mosaic-core/     (unchanged — existing library)
├── shire-cli/       (unchanged — headless dispatcher)
└── mosaic-tui/      NEW: ratatui binary, reads shire run dirs
```

## Dependencies added

- `ratatui` ≥ 0.29
- `crossterm` ≥ 0.28

(Both were anticipated in the original `mosaic_spec.md` but deferred during v0.1.)

## Build sequence

1. Warm-up: stderr wiring + rate_limit event + preview improvement (one commit each)
2. Scaffold `mosaic-tui` crate + dependencies
3. Event loop + terminal init/teardown
4. Run-dir reader (parse `summary.jsonl` → `Vec<TileState>`)
5. Tile grid widget (ratatui)
6. Focus pane + keybinding layer
7. Polish: help overlay, status bar, error handling on missing run dir

Each step gets its own commit so the overnight history is legible in the morning.

## Ground rules for overnight autonomy

- Commit and push frequently. Never leave uncommitted work at a stopping point.
- If a subagent fabricates a result, catch it before committing (verify independently).
- If a genuine blocker hits, stop on that component, document it, and move on to the next.
- Keep v0.1 fully intact — any changes that touch `mosaic-core` or `shire-cli` must not
  regress the 71 automated tests or the 10 offline smoke tests.
