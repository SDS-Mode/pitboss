# Agent Shire v0.3 — Hierarchical Smoke Test Procedure

**Purpose:** Verify the v0.3 hierarchical-orchestration implementation works end-to-end
before promoting `v0.3.0-pre` → `v0.3.0`. Exercises behaviors the automated test suite
can't fully cover: real `claude` subprocess running as a lead, real MCP server over a
Unix socket, real `spawn_worker` calls producing live worker subprocesses, real budget
accounting, and real Ctrl-C drain with multiple in-flight children.

**Estimated runtime:** 30–45 minutes of wall-clock, most of it waiting on real Claude API calls.
**Estimated cost:** ~$0.50–$1.50 in Anthropic API usage assuming the Haiku model.

## Part 1 — Offline tests (no API calls)

These tests don't invoke `claude` at all; they validate hierarchical-manifest schema
handling and exit-code semantics. Each should complete in <1s.

### 1.1 Validate — happy hierarchical path

```bash
# Setup a throwaway git repo with three small files to inspect
mkdir -p "$SHIRE_SMOKE_DIR/repo-a"
cd "$SHIRE_SMOKE_DIR/repo-a"
git init -q && git config user.email t@t.x && git config user.name t
echo "alpha content" > file-a.txt
echo "beta content"  > file-b.txt
echo "gamma content" > file-c.txt
git add . && git commit -q -m "init"
cd "$SHIRE_SMOKE_DIR"

cat > hierarchical-happy.toml <<EOF
[run]
max_workers = 4
budget_usd = 1.00
lead_timeout_secs = 900

[defaults]
model = "claude-haiku-4-5"

[[lead]]
id = "triage"
directory = "$SHIRE_SMOKE_DIR/repo-a"
prompt = "Inspect file-a.txt, file-b.txt, file-c.txt in this directory, spawn one worker per file to summarize it in one sentence, wait for all workers, then write a combined summary to summary.md and exit."
EOF

shire validate hierarchical-happy.toml
```
**Expect:** `OK (hierarchical) — lead='triage', max_workers=4, budget=$1.00`. Exit 0.
- [ ] Pass

### 1.2 Validate — rejects mixed `[[task]]` and `[[lead]]`

```bash
cat > bad-mixed.toml <<EOF
[[task]]
id = "t1"
directory = "$SHIRE_SMOKE_DIR/repo-a"
prompt = "p"

[[lead]]
id = "lead1"
directory = "$SHIRE_SMOKE_DIR/repo-a"
prompt = "lead prompt"
EOF

shire validate bad-mixed.toml
echo "exit=$?"
```
**Expect:** stderr mentions `cannot combine [[task]] and [[lead]]`. Exit 2.
- [ ] Pass

### 1.3 Validate — rejects `max_workers` without a lead

```bash
cat > bad-maxworkers-no-lead.toml <<EOF
[run]
max_workers = 4

[[task]]
id = "t1"
directory = "$SHIRE_SMOKE_DIR/repo-a"
prompt = "p"
EOF

shire validate bad-maxworkers-no-lead.toml
echo "exit=$?"
```
**Expect:** stderr mentions `max_workers is only valid with a [[lead]] section`. Exit 2.
- [ ] Pass

### 1.4 Validate — rejects `budget_usd = 0`

```bash
cat > bad-budget-zero.toml <<EOF
[run]
max_workers = 4
budget_usd = 0
lead_timeout_secs = 900

[[lead]]
id = "lead1"
directory = "$SHIRE_SMOKE_DIR/repo-a"
prompt = "p"
EOF

shire validate bad-budget-zero.toml
echo "exit=$?"
```
**Expect:** stderr mentions `budget_usd must be > 0`. Exit 2.
- [ ] Pass

### 1.5 Validate — rejects `max_workers = 17`

```bash
cat > bad-maxworkers-big.toml <<EOF
[run]
max_workers = 17
budget_usd = 1.00
lead_timeout_secs = 900

[[lead]]
id = "lead1"
directory = "$SHIRE_SMOKE_DIR/repo-a"

EOF

shire validate bad-maxworkers-big.toml
echo "exit=$?"
```
**Expect:** stderr mentions `max_workers must be between 1 and 16 inclusive`. Exit 2.
- [ ] Pass

## Part 2 — Real `claude` hierarchical happy path

Each of these spawns a real lead subprocess that in turn calls `spawn_worker` via MCP
to spawn real worker subprocesses. Keep prompts short and cheap. The lead uses Haiku;
workers inherit the same model unless the lead passes overrides.

**Re-enter the scratch dir:**
```bash
cd "$SHIRE_SMOKE_DIR"
which claude
```

### 2.1 Three-worker triage run

Reuse `hierarchical-happy.toml` from §1.1 (lead prompt spawns one worker per file
and waits for all, then composes `summary.md`).

```bash
shire dispatch hierarchical-happy.toml
echo "exit=$?"
```

**Expect during run:**
- `triage` lead appears first; progress or log output shows the lead is running.
- Within a few seconds the lead issues three `spawn_worker` calls; three new
  task entries appear in the run dir.
- Workers all complete, then the lead composes its combined summary and exits.
- Exit code 0.

**Expect after run:**
```bash
RUN_DIR=$(ls -td ~/.local/share/shire/runs/*/ | head -1)
echo "run: $RUN_DIR"
ls -la "$RUN_DIR"
ls "$RUN_DIR/tasks/"
```
- [ ] `manifest.snapshot.toml` present and matches the input manifest byte-for-byte.
- [ ] `resolved.json` present and parseable as JSON containing `lead.id = "triage"`.
- [ ] `meta.json` present, contains `shire_version: "0.3.0"` and a `claude_version` string (or null).
- [ ] `summary.jsonl` present with 4 lines (1 lead + 3 workers).
- [ ] `summary.json` present, `tasks_total: 4`, `tasks_failed: 0`, `was_interrupted: false`.
- [ ] `tasks/triage/stdout.log` present, non-empty.
- [ ] Three worker directories present under `tasks/` (names chosen by the lead).
- [ ] A `summary.md` file exists in the lead's worktree (or repo-a, if the lead ran without one).

### 2.2 Lead has `parent_task_id: null`, workers point at lead

```bash
jq '.tasks[] | {task_id, parent_task_id}' "$RUN_DIR/summary.json"
```
**Expect:**
- Exactly one record with `parent_task_id: null` and `task_id: "triage"`.
- Three records with `parent_task_id: "triage"`.
- [ ] Pass

### 2.3 Token usage is non-zero across the whole run

```bash
jq '[.tasks[].token_usage.input] | add' "$RUN_DIR/summary.json"
jq '[.tasks[].token_usage.output] | add' "$RUN_DIR/summary.json"
```
**Expect:** both totals > 0, typically hundreds-to-thousands for a simple triage.
- [ ] Pass

### 2.4 Lead `claude_session_id` is present (feeds §6)

```bash
jq '.tasks[] | select(.task_id == "triage") | .claude_session_id' "$RUN_DIR/summary.json"
```
**Expect:** a non-null string that looks like a session id. Take note — Part 6 will use
this implicitly via `shire resume`.
- [ ] Pass


### 3.2 Launch mosaic in terminal 2

```bash

```

**Expect (in mosaic):**
- [ ] The lead tile title shows `[LEAD] triage` (with a bolder border than worker tiles).
- [ ] As workers spawn, their tiles show a sub-line `← triage` under the title.
- [ ] Once all three workers are visible, the status bar at the top right shows

Press `q` in the mosaic TUI.
- [ ] Mosaic exits without hanging.
- [ ] Terminal 1's shire dispatch continues or has already exited 0.

## Part 4 — Budget enforcement

Force a very low budget so the lead's `spawn_worker` calls start being rejected
part-way through, and confirm the lead reacts gracefully (typically: composes a
partial summary with an apology rather than crashing).

### 4.1 Edit the manifest to `budget_usd = 0.05`

```bash
cd "$SHIRE_SMOKE_DIR"
cp hierarchical-happy.toml budget-tight.toml
# Edit by hand or use sed:
sed -i 's/budget_usd = 1.00/budget_usd = 0.05/' budget-tight.toml
grep budget_usd budget-tight.toml   # expect: budget_usd = 0.05
```

### 4.2 Run it

```bash
shire dispatch budget-tight.toml
echo "exit=$?"

RUN_DIR=$(ls -td ~/.local/share/shire/runs/*/ | head -1)
echo "run: $RUN_DIR"
```

**Expect:**
- The first one or two worker spawns may succeed; subsequent ones fail.
- The `tasks/triage/stdout.log` contains the string `budget exceeded` somewhere
  (that's the MCP tool's error returned to the lead).
- The lead acknowledges the budget limit in its final message rather than crashing.
- Exit code 0 or 1 depending on whether the lead itself errored, but **not** a panic
  and **not** a Rust-level crash.

```bash
grep -c 'budget exceeded' "$RUN_DIR/tasks/triage/stdout.log" || true
jq '.tasks[] | {task_id, status, exit_code}' "$RUN_DIR/summary.json"
```

- [ ] `budget exceeded` appears at least once in the lead's stdout log.
- [ ] Lead's `status` is `Success` (or the lead's final message gracefully mentions the budget).
- [ ] No Rust panic or unexpected SIGABRT in stderr.

## Part 5 — Ctrl-C drain during a hierarchical run

Manual, requires interactive terminal. Same interrupt semantics as v0.1, but now
with multiple worker children alive at once.

### 5.1 Start a run with longer prompts so you have time to interrupt

```bash
cat > long-triage.toml <<EOF
[run]
max_workers = 4
budget_usd = 2.00
lead_timeout_secs = 900

[defaults]
model = "claude-haiku-4-5"

[[lead]]
id = "triage-long"
directory = "$SHIRE_SMOKE_DIR/repo-a"
prompt = "Spawn three workers. Each worker should carefully analyze one of file-a.txt, file-b.txt, file-c.txt and write a 10-paragraph in-depth essay about it. After all workers finish, compile the essays into a single report.md file."
EOF

shire dispatch long-triage.toml
```

### 5.2 Wait ~5–10s for the lead to spawn at least one worker, then Ctrl-C once

**Expect (first Ctrl-C):** stderr emits the drain message. In-flight workers keep
running; no new spawn_worker calls are accepted (MCP returns a "shutting down"
error for any further spawn attempts).

- [ ] Drain message observed

Wait for the run to finalize (workers drain up to `TERMINATE_GRACE` = 10s).

### 5.3 Inspect the results

```bash
RUN_DIR=$(ls -td ~/.local/share/shire/runs/*/ | head -1)
jq '.was_interrupted, .tasks[] | .status' "$RUN_DIR/summary.json"
echo "exit=$?"   # from the most-recent shire dispatch above
```

**Expect:**
- `was_interrupted: true` in the summary.
- At least one task has `status: "Cancelled"`.
- Shell reports `exit=130`.
- [ ] `was_interrupted: true`
- [ ] Exit code 130

## Part 6 — `shire resume` after a hierarchical run

Re-run a completed hierarchical manifest with `shire resume`; the lead is respawned
with `--resume <session-id>` so Claude skips prior context and continues (or re-answers
idempotently for short prompts).

### 6.1 Use the successful run from §2.1

```bash
# Find the original run dir and its id
cd "$SHIRE_SMOKE_DIR"
ls -td ~/.local/share/shire/runs/*/ | head -3
ORIG_RUN=$(basename "$(ls -td ~/.local/share/shire/runs/*/ | head -1)")
echo "resuming: $ORIG_RUN"
```

Note: pick a clean successful run (from §2.1, not a Ctrl-C'd run from §5). If the
most recent run was an interruption, substitute its predecessor.

### 6.2 Resume

```bash
shire resume "$ORIG_RUN"
echo "exit=$?"
```

**Expect during run:**
- The lead spawns immediately.
- Exit code 0 (the lead completes its work again).

### 6.3 Confirm `--resume <session-id>` was used

Resume produces a **new** run directory. Find its lead stdout log and grep.

```bash
NEW_RUN_DIR=$(ls -td ~/.local/share/shire/runs/*/ | head -1)
echo "new run: $NEW_RUN_DIR"

# The spawn args are logged; look for --resume in the process spawn log or
# check resolved.json which records resume_session_id on the lead.
jq '.lead.resume_session_id' "$NEW_RUN_DIR/resolved.json"
```

**Expect:** a non-null UUID-like string that matches the `claude_session_id` from
the original run's summary.
- [ ] `lead.resume_session_id` is set in the new run's `resolved.json`.
- [ ] The string value equals the original run's `tasks[] | select(.task_id=="triage") | .claude_session_id`.

```bash
# Cross-check
jq -r '.tasks[] | select(.task_id=="triage") | .claude_session_id' \
  ~/.local/share/shire/runs/"$ORIG_RUN"/summary.json
jq -r '.lead.resume_session_id' "$NEW_RUN_DIR/resolved.json"
```
- [ ] Both values match.
