# v0.2.1 Roadmap (shipped)

**Status as of 2026-04-17 (overnight session):** all Tier 1 items + two of
three Tier 2 items shipped on `feat/v0.2.1`. Broadcast (Tier 2) deferred
pending interactive snap-in design work.

Sketch of the next increment on top of `v0.2-alpha` (Mosaic TUI read-only
observer). Prioritized by impact × implementation cost.

## Tier 1 — high-impact, self-contained

### Run picker (~2–4 hours) — **SHIPPED**

Lets the user switch between runs without relaunching `pitboss-tui`.

- New mode `Mode::PickingRun` in `state.rs`.
- New keybinding `o` (from Normal) → switch to PickingRun.
- `state::RunList` — Vec of `{run_id, started_at, tasks_total, tasks_failed, is_complete}`.
- Watcher reloads run list every few seconds when in PickingRun mode.
- Ratatui `List` widget. `j`/`k` navigate, `Enter` selects (swaps the watcher's
  run_dir), `Esc` cancels.
- On selection: send new `run_dir` to the watcher via a new channel, or restart
  the watcher thread. Simpler: kill old watcher and spawn a new one pointing at
  the new dir.

**Risk:** low. Entirely additive to existing state machine.

### Snap-in view-only (~3–5 hours) — **SHIPPED**

Press `Enter` on a Running tile to enter a full-screen "session view" showing the
complete stdout.log in real time. Still read-only — no stdin routing. This is
separate from the interactive snap-in described in the original Mosaic spec.

- New mode `Mode::SnapIn { task_id: String }`.
- Enter key from Normal: if focused tile is Running or recently Done, switch mode.
- Full-screen ratatui `Paragraph` with scroll state; auto-scroll to bottom when new
  lines arrive. `Esc` to exit.
- Watcher's focus channel already supports this — the TUI just changes how it
  renders the same data.

**Risk:** low-to-medium. Scroll/resize edge cases need care.

### SQLite SessionStore impl (~3–6 hours) — **SHIPPED**

v0.1 defined `SessionStore` trait and shipped `JsonFileStore`. Adding a
`SqliteStore` unlocks cross-restart resume and richer queries.

- New module `crates/pitboss-core/src/store/sqlite.rs`.
- `rusqlite` dependency, `bundled` feature (so consumers don't need system libsqlite).
- Schema:
  ```sql
  CREATE TABLE runs (
      run_id TEXT PRIMARY KEY,
      manifest_path TEXT NOT NULL,
      shire_version TEXT NOT NULL,
      claude_version TEXT,
      started_at TEXT NOT NULL,
      ended_at TEXT,
      tasks_total INTEGER,
      tasks_failed INTEGER,
      was_interrupted INTEGER DEFAULT 0
  );
  CREATE TABLE task_records (
      run_id TEXT NOT NULL,
      task_id TEXT NOT NULL,
      status TEXT NOT NULL,
      exit_code INTEGER,
      started_at TEXT, ended_at TEXT, duration_ms INTEGER,
      worktree_path TEXT, log_path TEXT,
      token_input INTEGER, token_output INTEGER,
      token_cache_read INTEGER, token_cache_creation INTEGER,
      claude_session_id TEXT, final_message_preview TEXT,
      PRIMARY KEY (run_id, task_id)
  );
  ```
- `init_run` does `CREATE TABLE IF NOT EXISTS` once; inserts the run.
- `append_record` upserts into task_records.
- `finalize_run` updates runs.ended_at and other aggregates.
- `load_run` joins and reconstructs `RunSummary`.
- Tests: mirror `JsonFileStore` integration tests with `tempfile::NamedTempFile`.

**Risk:** medium. `rusqlite`'s bundled feature pulls in a C dependency; cross-platform
builds need verification. Transactional semantics for append+finalize need a brief thought.

### `pitboss resume <run-id>` (~2–3 hours) — **SHIPPED**

Wires `claude --resume <session-id>` into `SessionHandle::spawn`.

- `resolve.rs::ResolvedTask` already records `branch`, `directory`, `prompt`.
- Extend `ResolvedTask` with `resume_session_id: Option<String>`.
- `spawn_args`: if `resume_session_id` set, include `--resume <id>` instead of `-p`.
- New CLI command: `pitboss resume <run-id>`:
  - Load prior run from store (works with either `JsonFileStore` or `SqliteStore`).
  - For each task in the prior run, construct a fresh `ResolvedTask` with
    `resume_session_id = claude_session_id from prior TaskRecord`.
  - Dispatch normally.
- What happens if a task had no session_id (SpawnFailed)? Skip, or retry from scratch.

**Risk:** medium. Claude's `--resume` behavior should be verified manually —
whether it accepts `-p "prompt"` alongside `--resume`, what happens if the session
is expired, etc. The smoke test should cover the happy path.

## Tier 2 — nice-to-have

### Cost estimation (~1 hour) — **SHIPPED**

Hard-coded price table per model. Show estimated USD cost per tile and in the
stats bar. Haiku ~$0.80/M input, Sonnet ~$3/M input, etc.

Risk: prices change. Keep the table in a separate `prices.rs` module for easy
updates. Lives in `crates/pitboss-tui/src/prices.rs`.

### Watch mode timer (~30 min) — **SHIPPED**

Show total elapsed wall-clock since first task started, not just sum of durations.
Requires tracking earliest `started_at` across all tiles. `AppSnapshot` now
carries `run_started_at: Option<DateTime<Utc>>` and the title bar shows
`— live Xm YYs` while any tile is Pending/Running.

### Broadcast (~2 hours)

`pitboss-tui b` followed by a string → sends the same prompt to all tiles simultaneously.
Requires the interactive snap-in path to already exist. Defer until after real
snap-in lands.

## Tier 3 — deferred again, with reasoning

### Interactive snap-in (keystroke passthrough)

This is the original Mosaic vision. It needs:

- Persistent bidirectional IPC with a running `claude` subprocess (currently claude
  in `-p` mode is one-shot and doesn't accept new input after the initial prompt).
- Either change pitboss to use interactive claude mode (no `-p`), or design a
  different control channel.
- Careful input mode management in the TUI (raw input, escape sequences, etc.).

This is a design project, not a feature project. The spec needs its own brainstorm
pass. It's **not** part of v0.2.x.

### Hierarchical lead/worker orchestration

Out of scope for everything in v0.x. Needs a control-channel design first.

## Suggested implementation order

1. **Run picker** — high visible value, low risk, builds TUI muscle.
2. **SQLite SessionStore** — foundational for everything that comes after.
3. **`pitboss resume`** — cashes in on SQLite immediately.
4. **View-only snap-in** — high visible value in TUI.
5. Cost estimation + elapsed timer (small polish items).

Full interactive snap-in deferred until a proper design pass.

## Ground rules for v0.2.1

- `v0.2-alpha` and `v0.2.0` (when tagged) should keep working unchanged. New
  features land behind env vars or new subcommands, not silent behavior changes.
- `cargo test --workspace --features pitboss-core/test-support` must stay green
  throughout. Target: every task in Tier 1 adds at least one integration test.
- Every interactive change gets a checkbox added to
  `docs/v0.2-tui-smoke-test.md`.
