# Contributing to Pitboss

Thanks for your interest in contributing. This file is the short
operator-facing guide; the deeper references are linked below.

## Where to look first

- **`README.md`** — what Pitboss is and how operators use it.
- **`AGENTS.md`** — the canonical, machine-targeted spec. Full manifest
  schema, MCP tool catalogue, migration tables, decision trees. **Read
  this before changing anything that touches the manifest schema, the
  MCP server, or the dispatcher's actor model.** It's bundled into the
  binary (`pitboss agents-md`) — the file IS the spec, not docs about
  the spec.
- **`RELEASE.md`** — full release process. Read if you're cutting a
  version.
- **`book/`** — the operator-facing mdBook (rendered at
  <https://sds-mode.github.io/pitboss/>).

## Building and testing

```bash
# Full workspace build + tests (~536 tests). The test-support feature
# exposes fixtures used across the workspace; always pass it.
cargo test --workspace --features pitboss-core/test-support

# Aliases (defined in .cargo/config.toml):
cargo lint      # clippy --workspace --all-targets --all-features -D warnings
cargo tidy      # fmt --all -- --check

# Offline smoke tests (no API calls — uses fakes from tests-support/).
scripts/smoke-part1.sh        # 10 offline flat-mode tests
scripts/smoke-part3-tui.sh    # 7 non-interactive TUI tests
scripts/ci-local.sh           # full local CI replica

# Web SPA (embedded into the pitboss-web binary by rust-embed at build time):
cd crates/pitboss-web/spa && npm install && npm run build
```

Toolchain is pinned to Rust 1.95.0 via `rust-toolchain.toml`. **Never
edit that file to "fix" CI failures** — fix the warning instead.

## Smoke-test setup (live-claude validation)

Most contributors won't need this; it's for end-to-end testing against
the real `claude` CLI. The dispatch test manifests reference target
project directories via local symlinks at the repo root. These symlinks
are `.gitignore`d so they don't end up in commits — wire them to your
own checkouts:

```bash
# Wherever your test target lives:
ln -s ~/your-projects/some-project pitboss_proj

# (Optional) a second target for testing concurrent dispatches:
ln -s ~/your-projects/another-project angrylarry
```

The symlinks must NOT be committed; check `.gitignore` if you see them
appearing in `git status`.

## Pull-request style

`main` is trunk-based and always shippable. Every PR squash-merges as a
single commit. **Your PR title becomes the commit subject — and the
CHANGELOG entry.** Get the prefix right and the rest is automatic.

Format: `type(scope): subject`

| Prefix | CHANGELOG section |
|---|---|
| `feat(scope): …` | `### Added` |
| `fix(scope): …` | `### Fixed` |
| `refactor(scope): …` / `perf(scope): …` | `### Changed` |
| `chore(deps): bump …` | `### Dependencies` |
| `docs:` / `test:` / `chore:` / `ci:` / `build:` / `style:` | excluded from CHANGELOG |

Breaking changes use the `!` marker:
`feat(mcp)!: rename spawn_worker.task_id to actor_id` — or a
`BREAKING CHANGE:` footer in the commit body. Cliff promotes those to
their own section.

Subject style:

- Imperative present tense — *add X*, not *added X* or *adds X*.
- No trailing period.
- ~70 chars max.
- Self-contained — a reader of the changelog should understand the
  change without opening the PR.
- **Don't include `(#NNN)`** — `git-cliff` adds a PR link automatically
  at release time.

Body conventions:

- Explain *why*, not *what* — the diff shows what changed.
- Reference issues with `Closes #NNN` / `Fixes #NNN` — GitHub auto-closes
  on merge.
- File paths and identifiers are useful for triage; prose belongs in
  the description, not in source comments.
- AI-assisted commits use a `Co-Authored-By:` footer with the model's
  noreply address (already conventional in this repo).

## Don't hand-edit `CHANGELOG.md`

CHANGELOG entries from v0.9.2 onward are generated by
[`git-cliff`](https://git-cliff.org/) at release time — see
`cliff.toml` for the config and `RELEASE.md` step 3 for the invocation.
Hand-editing CHANGELOG.md in feature PRs causes merge conflicts and
duplicates work the tool already does. If you need to add prose context
for a particularly significant entry, the release-driver can layer it
in at version-cut time, after cliff produces the bullet list.

## Reporting issues

- **Security vulnerabilities** — do NOT open a public issue. See
  `SECURITY.md` for the disclosure flow.
- **Bug reports** — include the run-id (look in
  `~/.local/share/pitboss/runs/`), the manifest, and the output of
  `pitboss --version`. The dispatcher writes everything you'd need to
  reproduce a failure under that run-id directory.
- **Feature requests** — open an issue describing the use case before
  the change. Pitboss aims to stay opinionated about the actor model;
  not every workflow is in scope.

## Project structure

| Crate | Role |
|---|---|
| `pitboss-cli` | `pitboss` binary — dispatch, validate, schema, init, resume, container-dispatch, status, list, prune. |
| `pitboss-tui` | `pitboss-tui` binary — read-only floor view: tile grid, log tailing, live policy editor. |
| `pitboss-web` | `pitboss-web` binary — axum + embedded SvelteKit SPA. SSE event streams, control writes, manifest wizard, insights dashboard. |
| `pitboss-core` | Shared types, sqlite store, stream-json parser, session/process/worktree modules. |
| `pitboss-schema` + `pitboss-schema-derive` | `#[field(label, help)]` proc-macro for manifest field metadata. |
| `tests-support/` | Fakes used across the workspace test suite (`fake-claude`, `fake-mcp-client`, `fake-control-client`). |

## Conventions worth knowing

- The web SPA's build artifacts (`crates/pitboss-web/spa/build/`) are
  baked into the `pitboss-web` binary by `rust-embed` at compile time.
  Run `npm run build` before `cargo install`.
- `AGENTS.md` is the source of truth, not docs about it. Changes that
  affect agent behavior must update `AGENTS.md` in the same PR.
- The manifest schema's `#[field(label, help)]` proc-macro is the
  single source of truth for field metadata. Don't hand-write field
  labels in the SPA — pull them from the schema.
- Per-run artifacts under `~/.local/share/pitboss/runs/<run-id>/` are
  consumed by the TUI, web console, and `pitboss status`. Schema
  changes there ripple to all three.
